## 文档边界

本文档定义 life-ledger 首个可开发版本的产品目标、MVP 范围、功能需求和验收标准。

本文档不替代 UI 设计、API 设计或数据模型设计：

- UI 结构和页面原型见 `docs/ui/`
- API 细节后续写入 `docs/dev/api-design.md`
- SQLite 表结构后续写入 `docs/dev/data-model.md`
- 阶段拆分写入 `docs/dev/task.md`

## 1. 项目概述

### 1.1 背景

个人生活中的重要日期、收支流水和关键决策通常分散在备忘录、聊天记录、表格和记账软件中，长期追踪、复盘和迁移成本较高。life-ledger 希望用一个本地优先的网页服务，把这些信息集中记录到一个可备份、可查询、可导入导出的个人数据库中。

### 1.2 项目目标

- 记录并管理重要日期、账单流水和关键决策。
- 通过三个一级页面完成核心操作：日期、账单、决策。
- 使用 SQLite 作为唯一数据源，保证数据可备份、可迁移。
- 支持账单 Excel 导入导出，降低从表格迁移和回退的成本。
- 支持单二进制部署，通过 `config.toml` 修改运行配置。

### 1.3 目标用户

首版目标用户是项目所有者本人，使用场景是个人自用，不面向公开注册或多人协作。

### 1.4 成功标准

- 用户可以通过一个二进制文件启动服务，并在浏览器中打开日期、账单、决策三个页面。
- 用户可以完成三类核心数据的新增、查看、编辑、删除。
- 用户可以下载账单 Excel 模板、导入 `.xlsx` 账单文件、导出当前账单数据。
- Excel 导入失败时可以看到具体行号和原因，并且数据库不产生部分写入。
- 服务重启后，已写入的数据仍可从 SQLite 中读取。
- 用户可以将服务部署在云服务器上，并通过 Caddy 反向代理访问。

### 1.5 核心约束

- 部署方式必须保持简单：单个二进制文件 + `config.toml` + 数据目录。
- 首版部署目标是云服务器 + Caddy 反向代理。
- 应用默认只监听 `127.0.0.1`，由 Caddy 负责对外 HTTPS 入口。
- 数据库使用 SQLite。
- 前端是 React 单页应用，最终由 Go `embed` 内嵌到二进制中。
- 不为首版引入 Redis、消息队列、微服务或外部迁移工具。

## 2. MVP 范围

### 2.1 首版页面

首版只包含三个一级页面：

- 日期页：`/important-dates`
- 账单页：`/transactions`
- 决策页：`/decisions`

根路径 `/` 重定向到 `/important-dates`。

### 2.2 首版包含

- 服务启动和配置文件读取。
- SQLite 初始化和内嵌 migration。
- 前端页面导航和三页基础界面。
- 重要日期基础 CRUD。
- 账单基础 CRUD。
- 账单 Excel 模板下载、导入、导出。
- 按月份和分类管理预算，并在账单页展示预算使用情况。
- 决策基础 CRUD。
- 决策状态管理：进行中、待复盘、已归档。
- 单用户访问控制。
- 登录设备记录和设备会话管理。
- 安全配置辅助命令：生成本地配置、密码哈希和会话密钥。
- 手动备份命令和恢复说明。

### 2.3 首版不包含

- 多用户注册和权限体系。
- 云同步。
- 银行卡、支付宝、微信等自动同步。
- 外部日历同步。
- 多币种。
- 提醒规则、通知能力和独立提醒中心页面。
- 移动端原生 App。
- AI 自动评分或决策建议。

## 3. 功能需求

### R1. 服务启动与配置读取

- **描述**：应用启动时读取 `config.toml`，根据配置启动 HTTP 服务、打开数据目录和 SQLite 数据库。
- **用户场景**：用户修改配置文件后，直接运行 `./life-ledger` 启动服务。
- **输入**：`config.toml`。
- **输出**：可访问的本地网页服务。
- **验收标准**：
  - Given 二进制文件所在目录存在合法 `config.toml`, When 用户运行 `./life-ledger`, Then 服务按配置监听并输出访问地址。
  - Given `config.toml` 缺失或格式非法, When 用户运行 `./life-ledger`, Then 程序启动失败并输出明确错误。
  - Given 数据目录不存在, When 用户运行 `./life-ledger`, Then 程序创建数据目录或明确说明无法创建的原因。
- **优先级**：P0
- **成熟度**：RS3

### R2. SQLite 初始化与数据持久化

- **描述**：应用启动时打开 SQLite，并执行内嵌 migration 初始化必要表结构。
- **用户场景**：用户首次启动服务后，无需手动创建数据库即可开始记录数据。
- **输入**：数据目录、数据库文件名、内嵌 SQL migration。
- **输出**：可读写的 SQLite 数据库。
- **验收标准**：
  - Given 数据库文件不存在, When 服务启动, Then 自动创建数据库并完成 schema 初始化。
  - Given 数据库已有旧 schema, When 服务启动, Then 只执行未执行过的 migration。
  - Given migration 执行失败, When 服务启动, Then 程序启动失败并输出失败的 migration 编号。
- **优先级**：P0
- **成熟度**：RS3

### R3. 前端三页导航

- **描述**：前端提供日期、账单、决策三个一级页面，并支持刷新、前进、后退保持当前页面。
- **用户场景**：用户在浏览器中切换三个核心功能模块。
- **输入**：浏览器路径。
- **输出**：对应页面内容。
- **验收标准**：
  - Given 用户访问 `/`, When 页面加载, Then 自动进入 `/important-dates`。
  - Given 用户访问 `/transactions`, When 页面刷新, Then 仍展示账单页。
  - Given 用户点击一级导航, When 路由切换, Then 页面内容和导航高亮同步变化。
- **优先级**：P0
- **成熟度**：RS3

### R4. 重要日期管理

- **描述**：用户可以记录生日、纪念日、证件到期、缴费日等重要日期。
- **用户场景**：用户希望集中查看未来一段时间内有哪些重要日期。
- **输入**：标题、日期、日期类型、重复规则、备注、标签。
- **输出**：重要日期列表和单条详情。
- **字段规则**：
  - 重复规则首版支持：不重复、每年、每月、每周。
  - 若重复规则为空，系统按不重复处理。
  - 若重复规则不是上述枚举值，系统拒绝保存并提示字段错误。
- **验收标准**：
  - Given 输入合法日期信息, When 用户保存, Then 系统创建日期记录并在列表中展示。
  - Given 必填字段缺失, When 用户保存, Then 系统拒绝保存并指出缺失字段。
  - Given 用户修改已有日期, When 保存成功, Then 列表和详情展示最新内容。
  - Given 用户删除日期, When 确认删除, Then 该记录不再出现在列表中。
- **优先级**：P0
- **成熟度**：RS3

### R5. 账单流水管理

- **描述**：用户可以记录收入和支出流水，并按时间、类型、分类、账户筛选。
- **用户场景**：用户希望集中管理个人收支记录，并保留后续统计基础。
- **输入**：日期、时间、类型、金额、分类、计入收支、计入预算、所属账本、对象、账户、标签、备注。
- **输出**：账单列表、账单详情、筛选结果。
- **字段规则**：
  - 类型首版仅支持 `收入` 和 `支出`。
  - `计入收支=是` 的账单参与收入、支出和余额统计。
  - `计入收支=否` 的账单仅作为流水记录保存，不参与收入、支出和余额统计。
  - `计入预算=是` 仅对类型为 `支出` 的账单生效。
  - 所属账本首版用于记录和筛选，不影响预算统计维度。
- **验收标准**：
  - Given 输入合法账单信息, When 用户保存, Then 系统创建账单并在流水列表中展示。
  - Given 金额为空、为零或格式非法, When 用户保存, Then 系统拒绝保存并提示金额错误。
  - Given 日期、时间、类型、金额、分类、计入收支、计入预算或所属账本缺失, When 用户保存, Then 系统拒绝保存并指出缺失字段。
  - Given 账单类型为支出且计入预算为是, When 用户保存账单, Then 该账单金额计入对应月份和分类的预算消耗。
  - Given 账单类型为收入或计入预算为否, When 用户保存账单, Then 该账单不计入预算消耗。
  - Given 用户设置筛选条件, When 查询账单, Then 列表只展示符合条件的记录。
  - Given 用户删除账单, When 确认删除, Then 该记录不再出现在流水列表中。
- **优先级**：P0
- **成熟度**：RS3

### R6. 账单 Excel 导入导出

- **描述**：用户可以下载账单模板、导入 `.xlsx` 账单文件，并导出数据库中的账单数据。
- **用户场景**：用户从已有表格迁移账单，或定期把账单导出为 Excel 备份。
- **输入**：`.xlsx` 文件、导出筛选条件。
- **输出**：导入结果、错误明细、导出的 `.xlsx` 文件。
- **验收标准**：
  - Given 用户请求模板, When 下载成功, Then 得到包含固定表头的 `.xlsx` 文件。
  - Given 用户查看模板, When 打开 Excel, Then 模板必填列表头为：日期、时间、类型、金额、分类、计入收支、计入预算、所属账本。
  - Given 用户查看模板, When 打开 Excel, Then 模板选填列表头为：对象、账户、标签、备注。
  - Given 用户上传非 `.xlsx` 文件, When 系统接收文件, Then 系统拒绝导入并提示仅支持 `.xlsx`。
  - Given 上传文件超过 5MB, When 系统接收文件, Then 系统拒绝导入并提示文件过大。
  - Given 上传 Excel 超过 5000 行, When 系统校验文件, Then 系统拒绝导入并提示超过行数上限。
  - Given 上传 Excel 包含多个工作表, When 系统解析文件, Then 只读取第一个工作表。
  - Given 上传 Excel 表头与模板不一致, When 系统校验文件, Then 系统拒绝导入并提示表头错误。
  - Given 必填列存在空值, When 系统校验文件, Then 系统拒绝导入并返回具体行号、列名和原因。
  - Given 类型不是 `收入` 或 `支出`, When 系统校验文件, Then 系统拒绝导入并返回具体行号、列名和原因。
  - Given 计入收支或计入预算不是 `是` 或 `否`, When 系统校验文件, Then 系统拒绝导入并返回具体行号、列名和原因。
  - Given 上传的 Excel 全部合法, When 用户确认导入, Then 系统在一个 SQLite transaction 中写入全部记录。
  - Given 上传的 Excel 存在错误行, When 用户导入, Then 系统返回具体行号和原因，并且不写入任何记录。
  - Given 导入校验失败, When 系统返回错误, Then 错误响应只包含行号、列名和错误原因，不回显整行原始内容。
  - Given 用户请求导出, When 导出成功, Then 得到可被 Excel 打开的 `.xlsx` 文件。
- **优先级**：P0
- **成熟度**：RS3

### R7. 决策记录管理

- **描述**：用户可以记录关键决策的背景、候选方案、最终选择、预期结果和复盘信息。
- **用户场景**：用户希望重要选择可以被回顾，而不是只留下最终结论。
- **输入**：标题、背景、结构化候选方案、最终选择、决策状态、复盘日期、复盘内容、标签。
- **输出**：决策列表、状态分组、决策详情。
- **验收标准**：
  - Given 输入合法决策信息, When 用户保存, Then 系统创建决策记录。
  - Given 用户添加候选方案, When 保存决策, Then 每个候选方案包含方案名称、优点、缺点和备注。
  - Given 决策尚未做出最终选择, When 用户查看列表, Then 该记录展示在进行中分组。
  - Given 决策已设置复盘日期且日期到达, When 用户查看决策页, Then 该记录展示为待复盘。
  - Given 用户完成复盘并归档, When 保存成功, Then 该记录进入已归档分组。
- **优先级**：P0
- **成熟度**：RS3

### R8. 基础统计

- **描述**：在页面内部展示轻量统计，不作为独立分析页。
- **用户场景**：用户在账单页快速查看收入、支出、结余和分类占比。
- **输入**：账单数据、筛选条件。
- **输出**：统计卡片和分类占比。
- **验收标准**：
  - Given 当前月份存在账单, When 用户打开账单页, Then 展示收入、支出、结余。
  - Given 用户设置筛选条件, When 统计刷新, Then 统计结果基于筛选后的账单集合。
  - Given 某账单的计入收支为否, When 系统计算收入、支出或余额, Then 该账单不参与统计。
- **优先级**：P1
- **成熟度**：RS2

### R9. 标签

- **描述**：重要日期、账单、决策可以使用标签辅助分类。
- **用户场景**：用户希望跨模块查找与家庭、健康、证件、学习等主题相关的记录。
- **输入**：标签名称、关联对象。
- **输出**：对象上的标签展示和后续筛选基础。
- **验收标准**：
  - Given 用户为记录添加标签, When 保存成功, Then 记录详情和列表展示该标签。
  - Given 用户按标签筛选, When 查询执行, Then 返回关联该标签的记录。
- **优先级**：P1
- **成熟度**：RS2

### R10. 单用户访问控制

- **描述**：系统必须提供应用内单用户登录。Caddy 只负责 HTTPS 和反向代理，不负责业务登录态。
- **用户场景**：用户将服务运行在云服务器上，通过 Caddy 反向代理访问，打开页面后先登录，再使用日期、账单和决策功能。
- **输入**：登录用户名、登录密码、`config.toml` 中的用户名和密码哈希。
- **输出**：登录态 Cookie、登录页、访问拒绝响应或退出登录结果。
- **验收标准**：
  - Given 用户未登录, When 访问日期、账单、决策页面, Then 系统展示登录页。
  - Given 用户未登录, When 请求受保护 API, Then 系统返回 401。
  - Given 用户输入正确凭据, When 登录, Then 系统设置 HttpOnly 登录态 Cookie 并允许访问页面和 API。
  - Given 用户输入错误凭据, When 登录, Then 系统拒绝登录并不创建登录态。
  - Given 用户点击退出登录, When 退出成功, Then 系统清除登录态并返回登录页。
  - Given `config.toml` 缺少用户名、密码哈希或会话密钥, When 服务启动, Then 程序启动失败并输出明确错误。
  - Given `config.toml` 中的会话密钥短于 32 字节, When 服务启动, Then 程序启动失败并提示会话密钥不符合安全要求。
- **优先级**：P0
- **成熟度**：RS3
- **备注**：Caddy Basic Auth 不作为首版访问控制方案。

### R11. 登录设备记录和会话管理

- **描述**：用户登录成功后，系统记录当前设备会话，并允许该设备在 7 天内免登录访问。用户可以查看和撤销已登录设备。
- **用户场景**：用户在自己的电脑或手机登录一次后，7 天内再次打开服务无需重新输入密码；如果发现陌生设备，可以在设备管理中撤销该设备。
- **输入**：登录成功事件、浏览器 User-Agent、客户端 IP、会话 Cookie、设备撤销操作。
- **输出**：登录设备列表、当前设备标识、设备管理入口、设备撤销结果、会话过期结果。
- **验收标准**：
  - Given 用户登录成功, When 系统创建登录态, Then 生成随机会话 token，Cookie 保存原始 token，SQLite 只保存 token 哈希。
  - Given 用户登录成功, When 系统记录设备, Then 记录设备名称、User-Agent、首次登录时间、最后访问时间、最后访问 IP、过期时间和撤销状态。
  - Given 同一设备在 7 天有效期内携带有效 Cookie, When 访问页面或 API, Then 系统允许访问且不要求重新登录。
  - Given 同一设备在 7 天有效期内多次访问服务, When 系统更新最后访问时间, Then 会话过期时间不被延长。
  - Given 会话超过 7 天, When 用户再次访问页面或 API, Then 系统要求重新登录。
  - Given 用户登录后打开账户菜单, When 点击登录设备, Then 系统展示设备管理视图。
  - Given 用户打开设备管理, When 页面加载, Then 展示当前有效设备、过期设备和已撤销设备。
  - Given 用户撤销非当前设备, When 撤销成功, Then 该设备的 Cookie 后续请求不再有效。
  - Given 用户撤销当前设备, When 撤销成功, Then 当前浏览器退出登录并返回登录页。
  - Given Cookie 中的 token 在数据库中不存在、已过期或已撤销, When 用户访问受保护资源, Then 系统拒绝访问并清除本地 Cookie。
- **优先级**：P0
- **成熟度**：RS3
- **备注**：7 天免登录采用固定过期时间，从登录成功时开始计算，不做滑动续期；再次登录会创建新的设备会话。设备管理不新增一级页面，放在登录后的账户菜单或弹窗中。

### R12. 登录失败限速

- **描述**：系统必须记录登录失败并进行限速，降低云服务器公网入口被暴力猜密码的风险。
- **用户场景**：攻击者或误操作在短时间内多次提交错误密码时，系统临时拒绝继续登录尝试。
- **输入**：登录用户名、客户端 IP、失败时间、失败原因。
- **输出**：登录失败记录、临时锁定状态、登录拒绝响应。
- **验收标准**：
  - Given 同一用户名或同一 IP 在 10 分钟内连续登录失败 5 次, When 再次尝试登录, Then 系统拒绝登录并提示稍后重试。
  - Given 用户名或 IP 被临时锁定, When 未到 15 分钟锁定期再次尝试登录, Then 系统不校验密码并直接拒绝。
  - Given 锁定期已过, When 用户再次尝试登录, Then 系统允许重新校验用户名和密码。
  - Given 用户登录成功, When 系统创建登录态, Then 清除该用户名和该 IP 的活跃失败计数。
  - Given 服务重启, When 用户再次尝试登录, Then 之前存储在 SQLite 中的未过期锁定状态仍然生效。
- **优先级**：P0
- **成熟度**：RS3
- **备注**：阈值和锁定时长作为 `config.toml` 配置项提供默认值；默认值为 10 分钟内 5 次失败后锁定 15 分钟。

### R13. 安全配置辅助命令和本地配置初始化

- **描述**：系统提供命令行辅助命令，用于生成本地 `config.toml`、bcrypt 密码哈希和安全的 `session_secret`，降低用户手写安全配置的出错概率。
- **用户场景**：用户首次部署到云服务器时，需要为 `config.toml` 生成 `password_hash` 和 `session_secret`。
- **输入**：明文密码输入、随机数源。
- **输出**：本地配置文件、bcrypt 密码哈希、随机 `session_secret`。
- **验收标准**：
  - Given 用户运行 `./life-ledger init-config`, When 二进制目录下不存在 `config.toml`, Then 程序创建 `config.toml`、`data/` 和 `backups/`，并输出一次性初始密码。
  - Given 二进制目录下已经存在 `config.toml`, When 用户运行 `./life-ledger init-config`, Then 程序拒绝覆盖并输出明确错误。
  - Given 用户运行 `./life-ledger hash-password`, When 按提示输入密码, Then 程序输出 bcrypt cost=12 的密码哈希。
  - Given 用户运行 `./life-ledger generate-secret`, When 命令执行成功, Then 程序输出至少 32 字节强随机生成的 `session_secret`。
  - Given 辅助命令执行失败, When 程序退出, Then 输出明确错误并使用非零退出码。
  - Given 用户运行辅助命令, When 命令执行, Then 不启动 HTTP 服务、不打开浏览器页面、不修改数据库。
- **优先级**：P0
- **成熟度**：RS3
- **备注**：`hash-password` 和 `generate-secret` 只输出配置值，不自动写入；`init-config` 只创建新配置，已存在时必须失败。

### R14. CSRF 防护

- **描述**：系统使用 Cookie 保存登录态，因此必须对会修改数据的请求启用 CSRF 防护，避免跨站请求伪造。
- **用户场景**：用户登录 life-ledger 后访问其他网站，恶意网站不能借助浏览器自动携带的 Cookie 发起有效写操作。
- **输入**：登录态 Cookie、CSRF token、HTTP 方法、请求头。
- **输出**：允许请求、403 拒绝响应或新的 CSRF token。
- **验收标准**：
  - Given 用户已登录, When 前端请求写操作 API, Then 请求必须携带 `X-CSRF-Token`。
  - Given 写操作 API 缺少 CSRF token, When 请求到达服务端, Then 系统返回 403。
  - Given 写操作 API 携带错误 CSRF token, When 请求到达服务端, Then 系统返回 403。
  - Given 写操作 API 携带正确 CSRF token, When 用户登录态有效, Then 系统继续处理业务请求。
  - Given 用户请求 `GET`、`HEAD` 或 `OPTIONS`, When 请求到达服务端, Then 系统不要求 CSRF token。
  - Given 用户退出登录, When 退出成功, Then 系统清除当前会话对应的 CSRF token。
- **优先级**：P0
- **成熟度**：RS3
- **备注**：CSRF token 与登录会话绑定，前端通过受保护的初始化接口或登录响应获取。

### R15. 敏感配置和文件权限

- **描述**：系统必须保护 `config.toml`、SQLite 数据库和日志中的敏感信息，避免云服务器部署时因权限过宽或日志泄露导致数据风险。
- **用户场景**：用户把服务部署到云服务器后，配置文件包含登录密码哈希和 `session_secret`，数据库包含个人生活数据，需要明确文件权限和日志脱敏规则。
- **输入**：`config.toml` 文件权限、数据目录权限、SQLite 数据库权限、日志内容。
- **输出**：启动成功、启动失败、权限错误提示或脱敏后的日志。
- **验收标准**：
  - Given `config.toml` 权限不是 `600`, When 服务启动, Then 程序启动失败并提示修正配置文件权限。
  - Given 数据目录权限比 `700` 更宽, When 服务启动, Then 程序启动失败并提示修正数据目录权限。
  - Given SQLite 数据库文件已存在且权限不是 `600`, When 服务启动, Then 程序启动失败并提示修正数据库文件权限。
  - Given SQLite 数据库文件不存在, When 服务创建数据库, Then 创建后的数据库文件权限应为 `600`。
  - Given 程序输出日志, When 日志包含敏感字段, Then 日志必须脱敏或省略敏感值。
  - Given 程序报告配置错误, When 错误涉及敏感配置项, Then 错误只说明字段缺失或非法，不输出字段实际值。
- **优先级**：P0
- **成熟度**：RS3
- **备注**：敏感字段包括明文密码、`password_hash`、`session_secret`、session token、Cookie、CSRF token 和 Excel 原始内容。

### R16. 安全事件审计

- **描述**：系统必须把关键安全事件和高风险操作作为结构化审计事件写入 SQLite，并可在应用日志中输出脱敏摘要，便于用户回溯登录、设备和敏感操作历史。
- **用户场景**：用户发现异常访问或误删数据时，可以查看关键事件发生时间、来源 IP、设备和操作结果。
- **输入**：登录事件、设备事件、Excel 导入事件、删除事件、配置安全检查事件。
- **输出**：`audit_events` 审计事件记录。
- **验收标准**：
  - Given 用户登录成功或失败, When 事件发生, Then 系统写入审计事件。
  - Given 用户退出登录, When 事件发生, Then 系统写入审计事件。
  - Given 系统创建或撤销登录设备, When 事件发生, Then 系统写入审计事件。
  - Given 用户执行 Excel 导入且成功或失败, When 事件完成, Then 系统写入审计事件。
  - Given 用户删除账单、重要日期或决策, When 删除成功, Then 系统写入审计事件。
  - Given 配置文件、数据目录或数据库权限检查失败, When 服务启动失败且数据库尚不可用, Then 系统向标准错误输出脱敏错误信息并以非零状态码退出。
  - Given 审计事件包含资源信息, When 系统写入事件, Then 只记录资源类型和资源 ID，不记录完整业务内容。
  - Given 审计事件包含失败原因, When 系统写入事件, Then 不记录密码、Cookie、session token、CSRF token 或 Excel 原始内容。
  - Given 系统输出应用日志, When 日志涉及审计事件, Then 只输出脱敏摘要，不输出敏感值或完整业务内容。
  - Given 审计事件已经写入应用日志, When 用户查看安全记录, Then 系统仍以 SQLite 中的 `audit_events` 作为事实来源。
- **优先级**：P0
- **成熟度**：RS3
- **备注**：审计事件必须写入 SQLite；应用日志可以输出脱敏摘要，但不能作为审计唯一来源。首版不做复杂审计后台，可在设备管理或后续设置入口提供只读查看能力。

### R17. CORS 和安全响应头

- **描述**：首版前端页面和 API 同源部署，不开放跨域 API，并设置基础安全响应头降低浏览器侧攻击面。
- **用户场景**：用户登录 life-ledger 后访问其他网站，其他网站中的 JavaScript 不能跨域读取 life-ledger API 响应。
- **输入**：HTTP 请求来源、HTTP 响应头。
- **输出**：同源 API 响应、跨域访问拒绝结果、安全响应头。
- **验收标准**：
  - Given 第三方网站发起跨域 API 请求, When 请求到达服务端, Then 系统不返回允许第三方读取响应的 CORS 头。
  - Given 任意 API 响应, When 响应返回浏览器, Then 不应包含 `Access-Control-Allow-Origin: *`。
  - Given life-ledger 自己的前端页面请求同源 API, When 请求到达服务端, Then 系统正常处理请求。
  - Given 页面或 API 响应返回浏览器, When 响应生成, Then 系统设置基础安全响应头。
- **优先级**：P0
- **成熟度**：RS3
- **备注**：基础安全响应头至少包括 `X-Frame-Options: DENY`、`X-Content-Type-Options: nosniff`、`Referrer-Policy: same-origin`。

### R18. 备份与恢复

- **描述**：系统提供手动备份命令生成本地备份包，并在部署文档中说明手动恢复流程。首版不提供自动恢复命令。
- **用户场景**：用户在云服务器上定期备份个人数据，或在迁移服务器前导出完整数据和配置。
- **输入**：当前 SQLite 数据库、`config.toml`、应用版本、数据库 schema 版本。
- **输出**：本地备份包和备份元数据。
- **验收标准**：
  - Given 用户运行 `./life-ledger backup`, When 备份成功, Then 系统生成包含 SQLite 数据库、`config.toml` 和 `backup-meta.json` 的备份包。
  - Given 系统生成 `backup-meta.json`, When 用户查看元数据, Then 元数据包含应用版本、备份时间和数据库 schema 版本。
  - Given 备份过程无法读取数据库或配置文件, When 备份失败, Then 程序输出明确错误并使用非零退出码。
  - Given 用户运行 `./life-ledger backup`, When 命令执行, Then 不启动 HTTP 服务、不修改数据库。
  - Given 用户需要恢复备份, When 查看部署文档, Then 文档说明停止服务、替换数据目录和配置文件、再启动服务的手动恢复流程。
- **优先级**：P0
- **成熟度**：RS3
- **备注**：备份包包含全部个人数据和登录能力，不默认上传云端，不默认加密；首版不提供 `restore` 命令。

### R19. 预算管理

- **描述**：用户可以按月份和分类设置预算，并在账单页查看预算使用情况。
- **用户场景**：用户希望控制每月不同分类的支出，例如餐饮、交通、学习等分类的月度预算。
- **输入**：月份、分类、预算金额。
- **输出**：预算列表、预算使用金额、剩余金额、使用比例。
- **验收标准**：
  - Given 用户输入合法月份、分类和预算金额, When 用户保存, Then 系统创建或更新该月份和分类的预算。
  - Given 预算金额为空、为零或格式非法, When 用户保存, Then 系统拒绝保存并提示预算金额错误。
  - Given 某月份某分类存在预算, When 账单页展示预算使用情况, Then 系统统计该月份、该分类、类型为支出且计入预算为是的账单金额。
  - Given 某账单类型为收入, When 系统计算预算消耗, Then 该账单不计入预算。
  - Given 某支出账单计入预算为否, When 系统计算预算消耗, Then 该账单不计入预算。
  - Given 用户修改或删除参与预算的账单, When 操作成功, Then 预算使用金额随最新账单数据重新计算。
  - Given 某分类预算已超支, When 用户查看预算使用情况, Then 系统展示超支状态。
- **优先级**：P0
- **成熟度**：RS3
- **备注**：首版预算维度为月份 + 分类，不按账户、对象或所属账本拆分。

## 4. 非功能需求

### 4.1 部署

- 构建产物应支持单二进制部署。
- 前端静态资源应被 Go `embed` 内嵌。
- 运行配置优先来自 `config.toml`。
- 命令行参数只保留 `--config` 这类必要入口。
- 首版部署在云服务器上，应用进程监听 `127.0.0.1`，由 Caddy 反向代理到应用端口。
- Caddy 负责公网入口、域名绑定和 HTTPS 证书。
- 应用只在请求来自可信反向代理地址时读取 `X-Forwarded-For`，默认可信反代地址为 `127.0.0.1`。
- 首版提供 `backup` 命令生成本地备份包，不提供自动 `restore` 命令。

### 4.2 数据一致性

- SQLite 是唯一数据源。
- Excel 只作为导入导出格式，不作为运行时数据源。
- 批量导入必须使用 transaction，失败时不能部分写入。
- migration 失败必须阻止服务启动。

### 4.3 性能

首版建议目标：

- 本地页面交互在常规个人数据量下无明显等待。
- 常规 API 请求在 5000 条以内数据量下 P95 小于 300ms。
- Excel 导入 5000 行以内应给出明确成功或失败结果。

### 4.4 安全

- 默认监听 `127.0.0.1`。
- 云服务器部署时，不允许应用进程直接监听公网地址作为默认配置。
- 通过 Caddy 对外暴露时，必须启用单用户访问控制。
- Caddy 负责 HTTPS、域名绑定和反向代理；应用负责登录页、会话 Cookie、API 访问保护和退出登录。
- `config.toml` 权限必须为 `600`，数据目录权限必须为 `700`，SQLite 数据库文件权限必须为 `600`；权限过宽时服务启动失败。
- 密码使用 bcrypt 哈希，默认 cost=12；密码不应明文存储在数据库或配置文件中，配置文件只保存密码哈希。
- 配置文件中出现明文密码配置时，服务必须启动失败并提示改用密码哈希。
- `session_secret` 用于签名登录 Cookie，必须来自 `config.toml`，长度至少 32 字节，缺失或太短时服务必须启动失败。
- 应提供 `init-config`、`hash-password` 和 `generate-secret` 辅助命令生成安全配置值。
- 登录态 Cookie 应使用 HttpOnly；云服务器 HTTPS 部署时应使用 Secure；SameSite 建议使用 Lax。
- 会话 token 必须使用密码学安全随机数生成，数据库只保存 token 哈希。
- 使用 Cookie 登录态时，所有非 `GET`、`HEAD`、`OPTIONS` 的业务 API 必须校验 CSRF token。
- 登录设备记录中可以保存 User-Agent 和最后访问 IP，但日志中不得打印原始 Cookie 或会话 token。
- 登录设备和登录失败限速使用解析后的客户端 IP：当请求来自可信反代时取 `X-Forwarded-For` 中第一个 IP，否则使用 TCP 连接的 RemoteAddr。
- 应用不得无条件信任客户端提交的 `X-Forwarded-For` 或 `X-Real-IP` 请求头。
- 登录设备的默认有效期为 7 天，过期或撤销后必须立即失效。
- 登录失败限速状态必须持久化到 SQLite，服务重启后未过期锁定仍应生效。
- 登录、退出、设备创建、设备撤销、Excel 导入、删除记录和配置安全检查失败必须写入 SQLite 结构化审计事件；应用日志可输出脱敏摘要。
- 首版不开放跨域 API，不返回 `Access-Control-Allow-Origin: *`。
- 页面和 API 响应必须设置基础安全响应头：`X-Frame-Options: DENY`、`X-Content-Type-Options: nosniff`、`Referrer-Policy: same-origin`。
- Excel 导入只接受 `.xlsx`，默认上传大小上限为 5MB，默认单次导入行数上限为 5000 行。
- Excel 导入只读取第一个工作表，表头必须匹配模板。
- 账单 Excel 必填列为日期、时间、类型、金额、分类、计入收支、计入预算、所属账本；选填列为对象、账户、标签、备注。
- Excel 上传文件不长期保存到磁盘；解析完成后应释放临时资源。
- Excel 导入错误响应只返回行号、列名和错误原因，不回显整行原始内容。
- 日志不得打印 Excel 原始内容。
- 日志不得打印明文密码、`password_hash`、`session_secret`、session token、Cookie 或 CSRF token。
- 审计事件不得记录明文密码、`password_hash`、`session_secret`、session token、Cookie、CSRF token、Excel 原始内容或完整业务记录内容。
- 应用日志不得作为审计唯一来源；安全记录查询以 SQLite `audit_events` 为准。
- 备份 `config.toml` 和 SQLite 数据库等同于备份登录能力和全部个人数据，必须在部署文档中提示用户妥善保存。
- 备份包不默认上传云端，不默认加密；用户需要自行妥善保存。
- 导入文件解析失败时应返回明确错误，不吞掉异常。

### 4.5 兼容性

- 桌面端优先支持现代 Chromium 系浏览器。
- 移动端需支持基础浏览和操作，但首版不追求原生 App 体验。
- 首版部署目标为 Linux 云服务器；文件权限检查按 Linux 权限模型实现。

## 6. 术语表

| 术语 | 定义 |
|------|------|
| Caddy | 部署在应用前面的反向代理，负责 HTTPS 终止和请求转发。 |
| SQLite | 应用本地数据库文件，用于保存业务数据、会话、设备和审计事件。 |
| CSRF | 跨站请求伪造防护；首版通过 CSRF token 保护非只读业务接口。 |
| CORS | 浏览器跨源访问控制；首版不开放跨源 API。 |
| 登录设备 | 用户成功登录后记录的一条设备会话，包含设备名称、最近访问时间、过期时间和撤销状态。 |
| 审计事件 | 用于记录安全相关行为的结构化事件，首版写入 SQLite。 |
| 计入收支 | 账单是否参与收入、支出和余额统计。 |
| 计入预算 | 支出账单是否参与预算消耗统计。 |
| 所属账本 | 账单归属的数据分组；首版用于记录和筛选，不影响预算维度。 |
| 重复规则 | 重要日期的重复方式；首版支持：不重复、每年、每月、每周。 |
